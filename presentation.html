PART 1

AI Literacy

AI Literacy는 AI를 읽고, 이해하고, 적절히 활용하는 능력입니다.

LLM이란

자주 사용되었던 프롬프팅 기법

LLM 기본 이해

LLM은 정답 DB가 아니라
문맥 기반 생성 모델입니다

잘하는 일

문맥을 읽고 자연스러운 문장을 생성합니다. 요약, 분류, 초안 작성에 강합니다.

조심할 일

자연스럽게 말해도 사실이 아닐 수 있습니다. 최신 정보와 내부 문서는 기본적으로 모릅니다.

서비스 관점

사용자가 AI 답변을 믿고 행동하려면, 답변의 근거와 검증 흐름이 함께 필요합니다.

대표 프롬프팅 기법

Zero-shot, Few-shot, CoT는
무엇일까요?

Zero-shot

예시 없이 지시만 제공하는 방식입니다.

예: "이 공지를 세 줄로 요약해줘."

Few-shot

입력과 출력 예시를 몇 개 보여줍니다. 분류 라벨이나 출력 형식을 맞출 때 유용합니다.

예: "A → 학사, B → 장학 형식처럼 다음 질문도 분류해줘."

CoT

Chain of Thought, 즉 단계적으로 생각하도록 유도하는 방식입니다. 과거 복잡한 추론 태스크에서 자주 사용됐습니다.

예전 예: "단계별로 생각해서 이유를 설명해줘."

프롬프팅 업데이트

요즘은 CoT 사용이 권장되지 않고,
few-shot도 필요한 경우에만!

예전 프롬프팅
  • 복잡하면 few-shot 예시를 계속 추가합니다.
  • 어려우면 "think step by step"을 붙입니다.
  • 이러한 방식은 긴 사고 과정을 출력하게 합니다.
최근 권장 흐름
  • 먼저 zero-shot으로 간결하게 시도합니다.
  • 필요할 때만 few-shot을 추가합니다.
  • 사고 과정보다 목표, 제약, 출력 형식을 중요시합니다.
공식 문서: OpenAI Reasoning best practices (CoT 프롬프트 회피, zero-shot 우선)

프롬프트 설계

좋은 프롬프트는
성공 방식과 실패 방식을 함께 정합니다

애매한 요청

"학교 규정에 대해 친절하고 정확하게 답해줘."

→ 무엇을 근거로 할지, 모르면 어떻게 할지, 어떤 형식으로 답할지 알 수 없습니다.

좋은 요청

"제공된 문서만 근거로 답하고, 근거가 없으면 확인 불가라고 말해줘. 답변의 출처도 표기해줘."

→ 역할, 제약, 출력 형식을 명확히 줍니다.

PART 2

RAG (Retrieval-Augmented Generation)

RAG는 모델이 기억만으로 답하지 않도록, 질문 시점에 관련 문서를 찾아 근거로 넣는 구조입니다.

비유하면 그냥 시험에서 Open-book 테스트로 바꾸는 일입니다.

RAG가 생긴 이유

어제 바뀐 공지를
모델이 알 수 있을까요?

모델 기억만 사용

모델은 학습한 내용과 입력 프롬프트 안에서 답합니다.

최신 공지, 사내 문서, 학교 규정처럼 계속 바뀌는 정보에 약합니다.

RAG 사용

질문 시점에 관련 문서를 검색해서 답변 근거로 넣습니다.

모델을 다시 학습시키지 않아도 지식 업데이트가 가능합니다.

RAG의 핵심 아이디어

Retrieve → Augment → Generate

Retrieve

사용자 질문과 관련된 문서 조각을 찾습니다.

Augment

찾은 문서를 프롬프트의 근거로 추가합니다.

Generate

모델이 근거를 바탕으로 답변과 출처를 생성합니다.

핵심은 "아는 척"이 아니라 "자료를 보고 답하게" 만드는 것입니다.

RAG Pipeline

문서를 찾아 답변에 넣는
전체 흐름입니다

문서 수집
청킹
임베딩·인덱싱
문서 조각 검색
리랭킹
프롬프트 반영
답변·출처

RAG는 벡터 DB 하나를 붙이는 작업이 아닙니다. 보시는 것처럼 문서 준비부터 답변·출처까지 이어지는 흐름입니다.

문서 준비: 수집과 청킹

AI가 참고할 자료실을 만들고
검색 가능한 조각으로 나눕니다

문서 수집

PDF, 웹페이지, Notion, DB, FAQ를 가져옵니다. 중복, 오래된 문서, 권한 없는 문서는 걸러야 합니다.

청킹

문서를 제목, 문단, 표, 조항 단위로 나눕니다. 너무 작으면 맥락이 사라지고, 너무 크면 잡음이 섞입니다.

임베딩·인덱싱·검색

인덱스를 만들고,
Hybrid Search로 찾습니다

Embedding

문서 조각과 사용자 질문을 숫자 벡터로 바꿉니다.

이 벡터 덕분에 표현이 달라도 뜻이 비슷한 문서를 찾을 수 있습니다.

Indexing

검색을 빠르게 하기 위해 벡터 인덱스와 키워드 인덱스를 만들어 둡니다.

벡터 인덱스는 의미에, 키워드 인덱스는 규정 번호와 고유명사에 강합니다.

Retrieval

질문이 들어오면 벡터 검색과 키워드 검색을 함께 쓰는 Hybrid Search로 후보 문서를 찾습니다.

이후 날짜, 학기, 권한 같은 메타데이터 조건으로 지금 답해도 되는 문서만 남깁니다.

리랭킹과 프롬프트 반영

찾은 문서를 다시 줄 세우고
답변 정책과 함께 넣습니다

Reranking

처음 검색된 후보 중 질문에 정말 답이 되는 문서를 위로 올립니다.

문서가 많아질수록 RAG 정확도를 좌우하는 핵심 장치입니다.

프롬프트 반영

상위 문서만 넣는 것이 아니라, 답변 정책까지 함께 전달합니다.

아래 context의 근거만 사용하세요. 근거가 부족하면 "확인 불가"라고 답하세요. 답변 끝에 사용한 docId를 표시하세요. [doc-17] ... [doc-22] ...

RAG 예시

질문: 휴학생도 계절학기를
신청할 수 있나요?

[doc-17] 계절학기 수강 신청 안내 재학생 및 휴학생 중 복학 예정자는 계절학기 수강 신청이 가능하다. [doc-22] 휴학/복학 규정 휴학생은 정규학기 수강 신청 대상이 아니다.
좋은 답변

가능할 수 있습니다. 다만 "휴학생 전체"가 아니라 "복학 예정자"라는 조건이 있습니다. 출처 [doc-17]

RAG의 장점과 주의점

RAG는 강력하지만
마법은 아닙니다

장점
  • 최신 문서와 내부 문서를 연결할 수 있습니다.
  • 모델 재학습 없이 지식을 업데이트할 수 있습니다.
  • 답변의 출처를 사용자와 운영자가 확인할 수 있습니다.
주의점
  • 청킹이 나쁘면 핵심 조건이 잘립니다.
  • 오래된 문서가 섞이면 답변이 흔들립니다.
  • 검색 품질과 답변 품질을 따로 평가해야 합니다.

PART 3

LLMOps

LLMOps는 LLM 기반 서비스를 배포한 뒤에도 품질, 비용, 안전성을 계속 관리하는 운영 방식입니다.

Ops는 Operations, 즉 운영을 뜻합니다.

DevOps의 운영 관점을 LLM 기반 기능으로 확장한 개념입니다.

Ops와 LLM 기반 서비스

LLM 기반 서비스는
챗봇보다 넓습니다

질문에 답하는 서비스

고객 상담, FAQ, 학사 규정 안내처럼 사용자 질문에 답합니다. 정답성, 출처, 모르는 질문 처리가 중요합니다.

업무 결과물을 만드는 서비스

문서 요약, 메일 초안, 코드 생성처럼 사람이 사용할 결과물을 만듭니다. 품질 기준과 검토 흐름이 필요합니다.

도구를 사용하는 서비스

검색, DB 조회, 예약, 승인 요청처럼 외부 도구를 호출합니다. 권한과 실패 처리가 중요합니다.

왜 LLMOps가 필요할까요?

LLM 서비스는 200 OK여도
틀린 답변일 수 있습니다

일반 API 예시

결제 API가 200 OK를 반환하고 주문이 저장되면, 성공 여부를 비교적 명확히 판단할 수 있습니다.

LLM API의 문제

답변이 틀리거나, 출처가 엉뚱하거나, JSON 형식이 깨지거나, 민감정보가 섞일 수 있습니다.

원인 추적 기준

어느 모델, 어느 프롬프트, 어느 문서 인덱스에서 문제가 생겼는지 추적해야 합니다.

로그와 운영 지표

나중에 원인을 찾을 수 있게
버전과 품질을 함께 봅니다

로그 예시
{ "model": "model-name", "promptVersion": "rag-qa-v3", "schemaVersion": "answer-v2", "retrievedDocIds": ["doc-17", "doc-22"], "latencyMs": 1840, "inputTokens": 1220 }

질문과 답변만 저장하면 왜 틀렸는지 찾기 어렵습니다. 모델, 프롬프트, 스키마, 검색 문서가 함께 남아야 합니다.

운영 지표
  • Quality: 정답률, 사용자 피드백
  • Grounding: 근거 문서에 충실한가
  • Cost/Latency: 비용과 응답 속도
  • Safety: 개인정보, 정책 위반, prompt injection

→ 정답률만 보지 않고, 근거성·비용·안전성을 함께 봅니다.

로그를 남기고, 평가하고,
조금씩 배포합니다

이 루프는 기록하고, 평가하고, 비교하고, 조금씩 배포한 뒤 문제가 생기면 되돌리는 흐름입니다.

LLMOps Loop 단계

각 단계에서는
무엇을 할까요?

Trace요청, 응답, 검색 문서, 모델과 프롬프트 버전을 기록합니다.
Evaluate정답성, 근거성, 스키마, 안전성을 평가합니다.
Compare새 버전이 기존 버전보다 나아졌는지 비교합니다.
Canary전체 배포 전 일부 사용자나 트래픽에 먼저 적용합니다.
Monitor품질, 비용, 지연시간, 사용자 피드백을 관찰합니다.
Rollback지표가 나빠지면 이전 버전으로 되돌립니다.
Improve실패 사례를 평가셋과 프롬프트 개선에 반영합니다.

Evals = Evaluations

자동 평가는
LLM 서비스의 테스트 코드입니다

Evaluation Set

Evaluation Set은 모델이나 프롬프트를 바꾸기 전에 반복해서 던져볼 질문 묶음입니다.

자주 들어오는 정상 질문뿐 아니라, 애매한 자격·날짜·권한 조건처럼 실수하기 쉬운 질문도 함께 넣습니다.

문서에 없는 질문, 오래된 문서, 충돌 문서도 포함해야 새 버전이 이전보다 나아졌는지 비교할 수 있습니다.

Pass Criteria

통과 기준은 결론이 맞는지에서 끝나지 않습니다. 답변이 출처 문서에 기반하는지, 출처와 출력 형식이 깨지지 않는지도 확인합니다.

또 개인정보나 prompt injection에 안전한지, 비용과 지연시간이 서비스 기준 안에 들어오는지도 함께 봅니다.
