PART 1

AI Literacy

AI Literacy는 AI를 읽고, 이해하고, 적절히 활용하는 능력입니다. 여기서는 LLM이 정답 DB가 아니라 생성 모델이라는 점과, 대표적인 프롬프팅 기법을 먼저 정리합니다.

LLM 기본 이해

LLM은 정답 DB가 아니라
문맥 기반 생성 모델입니다

잘하는 일

문맥을 읽고 자연스러운 문장을 생성합니다. 요약, 분류, 초안 작성에 강합니다.

조심할 일

자연스럽게 말해도 사실이 아닐 수 있습니다. 최신 정보와 내부 문서는 기본적으로 모릅니다.

서비스 관점

AI 답변이 사용자의 행동으로 이어지면, 근거와 검증 장치가 필요합니다.

대표 프롬프팅 기법

Zero-shot, Few-shot, CoT는
무엇일까요?

Zero-shot

예시 없이 지시만 제공합니다. "다음 글을 세 줄로 요약해줘"처럼 바로 요청하는 방식입니다.

Few-shot

입력과 출력 예시를 몇 개 보여줍니다. 분류 라벨이나 출력 형식을 맞출 때 유용합니다.

CoT

Chain of Thought, 즉 단계적으로 생각하도록 유도하는 방식입니다. 과거 복잡한 추론 태스크에서 자주 사용됐습니다.

프롬프팅 업데이트

요즘은 CoT를 습관적으로
요구하지 않습니다

예전 습관
  • 복잡하면 few-shot 예시를 계속 추가합니다.
  • 어려우면 "think step by step"을 붙입니다.
  • 긴 사고 과정을 출력하게 합니다.
최근 권장 흐름
  • 먼저 zero-shot으로 간결하게 시도합니다.
  • 필요할 때만 few-shot을 추가합니다.
  • 사고 과정보다 목표, 제약, 출력 형식을 명확히 줍니다.
공식 문서: OpenAI Reasoning best practices (CoT 프롬프트 회피, zero-shot 우선)

프롬프트 설계

좋은 프롬프트는
성공 방식과 실패 방식을 함께 정합니다

애매한 요청

"학교 규정에 대해 친절하고 정확하게 답해줘."

무엇을 근거로 할지, 모르면 어떻게 할지, 어떤 형식으로 답할지 알 수 없습니다.

서비스용 요청

"제공된 문서만 근거로 답하고, 근거가 없으면 확인 불가라고 말해줘. 답변에는 출처 docId를 포함해줘."

역할, 제약, 출력 형식을 명확히 줍니다.

PART 2

RAG (Retrieval-Augmented Generation)

RAG는 모델이 기억만으로 답하지 않도록, 질문 시점에 관련 문서를 찾아 근거로 넣는 구조입니다.

비유하면 closed-book 시험에서 open-book 시험으로 바꾸는 일입니다.

문서 준비, 검색, 리랭킹, 프롬프트 반영, 주의점이 하나의 파이프라인으로 연결됩니다.

RAG가 생긴 이유

어제 바뀐 공지를
모델이 알 수 있을까요?

모델 기억만 사용

모델은 학습한 내용과 입력 프롬프트 안에서 답합니다.

최신 공지, 사내 문서, 학교 규정처럼 계속 바뀌는 정보에 약합니다.

RAG 사용

질문 시점에 관련 문서를 검색해서 답변 근거로 넣습니다.

모델을 다시 학습시키지 않아도 지식 업데이트가 가능합니다.

RAG의 핵심 아이디어

Retrieve → Augment → Generate

Retrieve

사용자 질문과 관련된 문서 조각을 찾습니다.

Augment

찾은 문서를 프롬프트의 근거로 추가합니다.

Generate

모델이 근거를 바탕으로 답변과 출처를 생성합니다.

중요한 점은 "모델이 아는 척"하지 않고 "자료를 보고 답하게" 만드는 것입니다.

RAG Pipeline

문서를 찾아 답변에 넣는
전체 흐름입니다

문서 수집
청킹
임베딩·인덱싱
문서 조각 검색
리랭킹
프롬프트 반영
답변·출처

각 단계가 흔들리면 최종 답변도 흔들립니다.

문서 준비: 수집과 청킹

AI가 참고할 자료실을 만들고
검색 가능한 조각으로 나눕니다

문서 수집

PDF, 웹페이지, Notion, DB, FAQ를 가져옵니다. 중복, 오래된 문서, 권한 없는 문서는 걸러야 합니다.

청킹

문서를 제목, 문단, 표, 조항 단위로 나눕니다. 너무 작으면 맥락이 사라지고, 너무 크면 잡음이 섞입니다.

검색 준비와 검색

문서 조각을 의미 좌표로 저장하고
질문과 가까운 조각을 찾습니다

Embedding

텍스트를 숫자 벡터로 바꿔 의미적으로 가까운 문장을 찾게 합니다.

Hybrid Search

의미 검색은 개념에 강하고, 키워드 검색은 정책 번호와 고유명사에 강합니다.

Metadata Filter

학기, 부서, 날짜, 권한 같은 조건으로 후보 문서를 줄입니다.

리랭킹과 프롬프트 반영

찾은 문서를 다시 줄 세우고
답변 정책과 함께 넣습니다

Reranking

처음 검색된 후보 중 질문에 정말 답이 되는 문서를 위로 올립니다.

문서가 많아질수록 RAG 정확도를 좌우하는 핵심 장치입니다.

아래 context의 근거만 사용하세요. 근거가 부족하면 "확인 불가"라고 답하세요. 답변 끝에 사용한 docId를 표시하세요. [doc-17] ... [doc-22] ...

RAG 예시

질문: 휴학생도 계절학기를
신청할 수 있나요?

[doc-17] 계절학기 수강 신청 안내 재학생 및 휴학생 중 복학 예정자는 계절학기 수강 신청이 가능하다. [doc-22] 휴학/복학 규정 휴학생은 정규학기 수강 신청 대상이 아니다.
좋은 답변

가능할 수 있습니다. 다만 "휴학생 전체"가 아니라 "복학 예정자"라는 조건이 있습니다.

좋은 RAG 답변은 결론, 조건, 출처를 함께 보존합니다. [doc-17]

RAG의 장점과 주의점

RAG는 강력하지만
마법 버튼은 아닙니다

장점
  • 최신 문서와 내부 문서를 연결할 수 있습니다.
  • 모델 재학습 없이 지식을 업데이트할 수 있습니다.
  • 답변의 출처를 사용자와 운영자가 확인할 수 있습니다.
주의점
  • 청킹이 나쁘면 핵심 조건이 잘립니다.
  • 오래된 문서가 섞이면 답변이 흔들립니다.
  • 검색 품질과 답변 품질을 따로 평가해야 합니다.

PART 3

LLMOps

LLMOps는 LLM 기반 서비스를 배포한 뒤에도 품질, 비용, 안전성을 계속 관리하는 운영 방식입니다.

DevOps가 "만든 서비스를 잘 굴러가게 하는 일"이라면, LLMOps는 그 대상을 AI 기능으로 확장한 것입니다.

Ops와 LLM 기반 서비스

단순 챗봇만
LLM 서비스가 아닙니다

Ops

배포, 모니터링, 장애 대응, 비용 관리, 롤백처럼 "잘 굴러가게 하는 일"입니다.

LLM 서비스

챗봇, 사내 문서 검색, 코드/문서 Copilot, 데이터 분석 보조를 포함합니다.

Agent Workflow

검색, 도구 호출, 예약, 승인 요청처럼 여러 단계를 수행하는 AI 흐름도 포함됩니다.

왜 LLMOps가 필요할까요?

LLM 서비스는 200 OK여도
틀린 답변일 수 있습니다

일반 API

응답 성공, 상태 코드, latency만 봐도 많은 문제를 발견할 수 있습니다.

LLM API

호출은 성공했지만 답변이 틀리거나, 출처가 엉뚱하거나, JSON 형식이 깨질 수 있습니다.

복합 버전

모델, 프롬프트, 스키마, 검색 인덱스가 함께 바뀌므로 추적이 필요합니다.

로그와 운영 지표

나중에 원인을 찾을 수 있게
버전과 품질을 함께 봅니다

{ "model": "model-name", "promptVersion": "rag-qa-v3", "schemaVersion": "answer-v2", "retrievedDocIds": ["doc-17", "doc-22"], "latencyMs": 1840, "inputTokens": 1220 }
운영 지표
  • Quality: 정답률, 사용자 피드백
  • Grounding: 근거 문서에 충실한가
  • Cost/Latency: 비용과 응답 속도
  • Safety: 개인정보, 정책 위반, prompt injection

LLMOps Loop 단계

각 단계에서는
무엇을 할까요?

Trace요청, 응답, 검색 문서, 모델과 프롬프트 버전을 기록합니다.

Improve실패 사례를 평가셋과 프롬프트 개선에 반영합니다.

평가 케이스

정답이 있는 질문뿐 아니라, 문서에 없는 질문과 오래된 문서가 섞인 질문도 포함합니다.

통과 기준

정답, 출처, 불확실성, 스키마, 비용, 지연시간을 함께 확인합니다.

모를 때 모른다고 말하는지도 평가해야 합니다.
